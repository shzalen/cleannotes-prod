R3-P02 中 handleLogout 未清理 Pinia store 数据 — 跨用户数据残留风险

文件：src/App.vue (handleLogout, L95-107)

问题：handleLogout() 清理了 JWT、BroadcastChannel、memoStorage 定时器，但未清理 6 个 Pinia store 的响应式数据（taskStore / growthStore / aiStore / todoStore / memoStore / weeklyReportStore）。

由于 App.vue 是根组件，onMounted 仅执行一次。登出后重新登录（不刷新页面）时：

  • onMounted 不会再次执行 → store 不会重新加载
  • 无 watch(auth.isAuthenticated) 监听认证状态变化
  • 新用户会看到前一个用户的任务、备忘录、待办等数据，直到手动刷新页面

影响：同一标签页内切换用户时存在跨用户数据泄露风险。

修复建议：在 handleLogout 中调用各 store 的 $reset() 或手动清空数据。或在 onLoginSuccess 中调用 switchUser + Promise.all([store.load(true)]) 强制重新加载。最简方案：登录成功后 window.location.reload()。
R3-P01 中 memoStorage 事件监听器泄漏 — 重复添加

文件：src/services/memoStorage.ts (initRetryListeners, L37-59)

问题：initRetryListeners() 使用匿名函数注册 online 和 visibilitychange 事件监听器。cleanupMemoStorage() 清理了 setInterval 并重置 listenersInitialized = false，但未移除这两个事件监听器。

重新登录时 setMemoUserId() → initRetryListeners() 再次执行，因 listenersInitialized 已被重置为 false，会添加新的监听器，导致重复累积：

  • 1 次登出/登录 → 2 组监听器
  • N 次循环 → N+1 组监听器
修复建议：将匿名函数提取为具名函数引用，在 cleanupMemoStorage 中 removeEventListener。
R3-P03 中 Supabase 查询未使用 select= 限制返回字段

文件：src/services/supabase.ts (supabaseGetMemos, ~L120)

问题：memos 列表查询 ?user_id=eq.{uid}&order=created_at.desc 未使用 select= 参数，返回全部字段包括 content（可能包含数 MB 的 base64 图片数据）。

列表视图仅需 title / tags / pinned / icon / created_at / updated_at 等元数据，完整 content 仅在编辑时需要。

影响：首次加载和全量刷新时网络传输量显著增大，尤其对有大量富文本备忘录的用户。

修复建议：列表查询添加 &select=id,title,tags,pinned,icon,created_at,updated_at,content_preview，编辑时按需获取完整 content。或使用 PostgREST 的 columns 参数。
R3-S01 低 supabaseClient.ts localStorage key 硬编码项目 ID

文件：src/services/supabaseClient.ts (L48)

问题：getCurrentUserIdSync() 中 localStorage.getItem('sb-ghkyhbxltdxhkhpqltdr-auth-token') 硬编码了 Supabase 项目引用 ID。如果项目迁移到新的 Supabase 实例，此 key 将不匹配。

修复建议：从 SUPABASE_URL 动态推导 key 名称，或使用 supabaseClient.auth.getSession() 异步获取。
R3-P04 低 closeCrossTabSync 登出后破坏入站跨标签页同步

文件：src/App.vue (handleLogout) + src/services/crossTabSync.ts

问题：closeCrossTabSync() 清除了 listeners 数组并关闭 channel。App.vue 的 onMounted 中注册的 onCrossTabSync 回调被清除后，因 App.vue 是根组件不会重新 mount，重新登录后入站跨标签页同步不会恢复。

出站广播仍可工作（broadcastChange 会通过 ensureChannel 重建 channel），但本标签页不再接收其他标签页的更新通知。

修复建议：不在 logout 时调用 closeCrossTabSync()，或将跨标签同步监听注册移到 watch(auth.isAuthenticated) 中。
R3-P05 低 useTheme mediaQuery 监听器未清理

文件：src/composables/useTheme.ts

问题：matchMedia('(prefers-color-scheme: dark)') 的 change 事件监听器在 composable 中注册但未在组件销毁时移除。由于 useTheme 在 App.vue 中调用且 App.vue 是根组件，实际影响较小。

修复建议：返回 onCleanup 或在 onUnmounted 中 removeEventListener。
R3-P06 低 多个短时 setTimeout 未在组件销毁时清理

文件：多个组件（GreetingCard / TodayProgress / CountdownTimer 等）

问题：部分组件使用 setTimeout 实现延迟效果（如渐入动画、倒计时更新），但未在 onUnmounted 中 clearTimeout。由于超时时间通常很短（1-5 秒），实际泄漏风险极低。

修复建议：统一使用 useTimeoutFn 或手动存储 timer ID 并在 onUnmounted 清理。
R3-P07 低 onTaskDoneCallback 未在 logout 时清除

文件：src/stores/task.ts

问题：onTaskDoneCallback 回调函数在注册后不会在登出时清除。如果回调引用了已销毁组件的响应式数据，可能导致空引用错误。

修复建议：在 handleLogout 中调用 taskStore.clearCallback() 或在 store 中添加 $reset 逻辑。
R3-P08 低 memoStore.reorderMemo 逐条 upsert (N+1)

文件：src/stores/memo.ts (reorderMemo)

问题：拖拽排序时对每个移动的 memo 逐条调用 upsertMemo，产生 N 次网络请求。对于大量 memo 的重排可能较慢。

修复建议：使用 PostgREST 的批量 PATCH 或 RPC 函数一次性更新所有排序字段。
R3-P09 低 查询无 limit 参数

文件：src/services/supabase.ts (多个查询函数)

问题：大多数查询未添加 limit= 参数。虽然当前用户数据量通常不会太大，但理论上无上限的查询可能在数据量增长时影响性能。

修复建议：为任务/备忘录/待办列表查询添加合理的 limit=500 或实现分页。
R3-P10 低 vite.config manualCharts @tiptap/extension-mention 死代码

文件：vite.config.ts (L110-111)

问题：MemoMention.ts 已在第二轮修复中删除，但 manualChunks 中 @tiptap/extension-mention 和 @tiptap/suggestion 的分块条件仍保留。这些包可能已不在 bundle 中。

修复建议：删除 manualChunks 中对应的 mention/suggestion 条件。无害但属于死代码。
