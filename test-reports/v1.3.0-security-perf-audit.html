严重 安全 跨标签页同步缺失 — todo / weeklyReport / growth 数据不同步 S/P-01 ▶
src/stores/todo.ts · src/stores/weeklyReport.ts · src/stores/growth.ts

App.vue 监听了 5 种 BroadcastChannel 消息，但只有 task.ts 和 memo.ts 实际调用了 broadcastChange()。todo、weeklyReport、growth 三个 store 的所有写操作（add/update/delete/generate）从不广播变更。

影响：用户在标签页 A 添加一个待办事项，标签页 B 不会刷新，用户以为没保存。growth（养成系统）XP 变化也不会跨标签页同步。

缺失广播的方法：

todo.ts:       addTodo(), updateTodo(), removeTodo(), markConverted()
weeklyReport:  generateReport(), generateAiSummary(), deleteReport()
growth.ts:     applyXp(), checkAchievements(), refreshDailyState()

建议修复 在上述方法的末尾添加 broadcastChange('todos-updated') / broadcastChange('reports-updated') / broadcastChange('growth-updated')。

严重 性能 备忘录搜索使用 DOMParser — O(n) 逐条解析 HTML，性能极差 P-01 ▶
src/stores/memo.ts:14-18, 41-55

stripHtml() 使用 new DOMParser().parseFromString(html, 'text/html') 提取纯文本。在 filteredMemos computed 中，搜索时对每个 memo都调用此函数。

function stripHtml(html: string): string {
  if (!html) return ''
  const doc = new DOMParser().parseFromString(html, 'text/html')
  return doc.body.textContent || ''
}

// filteredMemos — 每次搜索框输入都触发
list = list.filter(m =>
  m.title.toLowerCase().includes(q) ||
  stripHtml(m.content).toLowerCase().includes(q) ||  // ← 100 个 memo = 100 次 DOMParser
  m.tags.some(t => t.toLowerCase().includes(q))
)

影响：100 条备忘录 × 每次按键 = 100 次 DOMParser 调用，搜索明显卡顿。

建议修复 缓存纯文本摘要（在 memo 加载/保存时计算 plainText 字段），或使用正则 html.replace(/<[^>]*>/g, '') 替代 DOMParser。

严重 性能 RichTextEditor dragover 事件监听器内存泄漏 P-02 ▶
src/components/RichTextEditor.vue:1315, 1364

addEventListener 和 removeEventListener 传入了不同的函数引用，导致监听器永远无法被移除。每次编辑器组件挂载/卸载都泄漏一个 dragover 监听器。

// onMounted (line 1315) — 添加匿名函数
rteWrapperRef.value?.addEventListener('dragover', (e: DragEvent) => {
  if (e.dataTransfer?.types.includes('Files')) {
    e.preventDefault()
  }
})

// onBeforeUnmount (line 1364) — 传入不同的空函数！
rteWrapperRef.value?.removeEventListener('dragover', (e: any) => {})

建议修复 将监听器提取为具名函数引用，add 和 remove 使用同一引用：

const handleDragOver = (e: DragEvent) => {
  if (e.dataTransfer?.types.includes('Files')) e.preventDefault()
}
// onMounted:
rteWrapperRef.value?.addEventListener('dragover', handleDragOver)
// onBeforeUnmount:
rteWrapperRef.value?.removeEventListener('dragover', handleDragOver)
高 安全 CSP 包含 'unsafe-eval' — 削弱 XSS 防护 S-02 ▶
index.html:7

CSP script-src 包含 'unsafe-eval'，允许 eval()、new Function() 等动态代码执行。虽然源码中未直接使用 eval，但 Mermaid 或 markmap 可能内部依赖它。

<meta http-equiv="Content-Security-Policy" content="
  default-src 'self';
  script-src 'self' 'unsafe-eval';   ← 削弱 XSS 防护
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: blob: https:;
  connect-src 'self' https:;
  ..." />

建议修复 调查哪个依赖需要 unsafe-eval，尝试替换为 nonce/hash 方案。如果 Mermaid 需要，可考虑仅在 Mermaid 渲染时动态注入 CSP 例外。

高 安全 Link 扩展 URL 验证不足 — 仅阻止 javascript: 前缀 S-03 ▶
src/components/RichTextEditor.vue:1230

Tiptap Link 扩展的 validate 函数仅检查 javascript: 前缀，可被多种方式绕过：

LinkExtension.configure({
  validate(url) { return !url.toLowerCase().startsWith('javascript:') }
})

// 绕过方式：
// data:text/html,
// vbscript:alert(1)
//  javascript: (前导空格)
// java\tscript: (制表符)
// javascript: (HTML 实体编码)

建议修复 使用白名单验证，仅允许安全协议：

validate(url) {
  try {
    const u = new URL(url)
    return ['http:', 'https:', 'mailto:', 'tel:'].includes(u.protocol)
  } catch {
    return false
  }
}
高 安全 登出 fire-and-forget — 缓存 JWT 未及时清除 S-04 ▶
src/stores/auth.ts:100-104 · src/services/auth.ts:81-84

logout() 未 await authSignOut()，而 setCachedAccessToken(null) 在 signOut() 内部 await 之后才执行。如果 signOut 的网络请求失败，缓存的 JWT 永远不会被清除。

// stores/auth.ts
function logout() {
  authSignOut() // fire and forget — 未 await!
  user.value = null
}

// services/auth.ts
export async function signOut(): Promise<void> {
  await supabaseClient.auth.signOut()  // 网络请求
  setCachedAccessToken(null)           // 只有上面成功才会执行
}

建议修复 在 logout() 中立即清除缓存 token，不依赖网络请求：

function logout() {
  setCachedAccessToken(null)  // 立即清除
  authSignOut().catch(() => {})  // 后台执行
  user.value = null
}

高 安全 旧 RLS 策略使用 x-user-id header — 可被伪造 S-06 ▶
supabase-schema.sql:96-141 · migration-auth-supabase.sql:47-89

原始 schema 使用 x-user-id HTTP header 进行 RLS 过滤，任何用户都可以在请求中设置任意 x-user-id 来冒充其他用户。迁移文件已将策略改为 auth.uid()，但：

  • 部分迁移 SQL（todos、weeklyReports、memos）仍使用旧的 x-user-id 方案
  • hybrid.ts 第 8 行注释仍写着"所有请求带 x-user-id header"，代码与注释不同步
  • 如果迁移顺序不正确，数据库可能仍存在 x-user-id 策略

建议修复 在 Supabase Dashboard → SQL Editor 中执行检查：SELECT tablename, policyname, cmd, qual FROM pg_policies WHERE schemaname='public' AND tablename LIKE 'cleannote_%';，确认所有策略都使用 auth.uid()。
