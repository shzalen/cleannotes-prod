发现的问题 10 项

R4-P01 登出时 Growth 数据未刷入云端（数据丢失风险） 中风险

问题描述：handleLogout() 在登出前调用了 flushPendingWrites()（备忘录）和 flushTaskWrites()（任务），但未调用 flushGrowthToCloud()（成长数据）。Growth 数据使用 2 秒防抖上传策略，如果用户完成任务（触发 XP 计算）后立即登出，2 秒内的 XP 事件、成就解锁、状态变更将丢失。

文件位置：src/App.vue:95-108

影响：用户完成最后一个任务后立刻登出 → XP 不记录 → 成就可能不解锁 → 等级不提升

修复建议 在 handleLogout() 中 auth.logout() 之前添加 await flushGrowthToCloud() 调用。需从 useGrowthStore() 解构 flushGrowthToCloud。
R4-P02 task.ts 模块级 visibilitychange 监听器使用匿名函数 低风险

问题描述：task.ts 第 92 行在模块顶层使用匿名函数注册 visibilitychange 事件监听器，无法通过 removeEventListener 移除。

文件位置：src/stores/task.ts:91-94

实际影响：低。登出时 window.location.href = '/login' 触发整页重载，监听器随页面销毁自然清理。但代码质量上不规范，且如果未来改为 SPA 路由登出则会产生泄漏。

修复建议 提取为具名函数引用，添加 export function cleanupTaskListeners() 导出函数（与 clearOnTaskDone() 类似），在 handleLogout 中调用。
R4-P03 ToggleExtension 事件监听器使用匿名函数 低风险

问题描述：ToggleExtension.ts 在 summary 元素上使用匿名函数注册 blur 和 keydown 事件，无法显式移除。

文件位置：src/extensions/ToggleExtension.ts:105, 114

实际影响：极低。这些监听器绑定在 Tiptap 编辑器内部的 DOM 元素上，当 editor.destroy() 执行时随 DOM 树一起销毁。但无法显式清理仍是不规范。

修复建议 提取为具名函数，在 extension 的 destroy() 回调中 removeEventListener。优先级低，可在下次编辑器重构时一并处理。
R4-P04 auth store onAuthStateChange 订阅未显式取消 低风险

问题描述：auth.ts store 中 unsubAuth 变量保存了 onAuthStateChange() 返回的取消订阅函数，但该函数从未被调用。Supabase Auth 状态监听订阅在页面生命周期内持续存在。

文件位置：src/stores/auth.ts:25, 30-38

实际影响：低。if (!unsubAuth) 守卫防止重复注册，且 window.location.href 整页重载会自然销毁订阅。但代码中存储了取消函数却从不使用，属于未完成的生命周期管理。

修复建议 在 Pinia store 的 $dispose 或 App.vue 的 onUnmounted 中调用 unsubAuth?.()。或在 logout() 函数中调用。
R4-P05 MemoView deep watch 性能开销 低风险

问题描述：MemoView.vue 第 102 行对选中备忘录使用 watch(..., { deep: true })。备忘录 content 字段可能包含数 MB 的 HTML（含 base64 图片或富文本），deep watch 会递归遍历所有嵌套属性。

文件位置：src/views/MemoView.vue:102

实际影响：低。该 watch 用于触发自动保存（800ms 防抖），功能上必要。但 deep watch 对大对象的性能开销是 O(n)，每次 content 变更都触发完整遍历。

修复建议 考虑改用 watch(() => selectedMemo.value?.updated_at, ...) 或在 content 变更时显式设置 dirty flag，避免 deep 遍历。需要同时监听 title/tags/pinned 等字段变更，可用多个浅 watch 替代。
R4-S01 console_sync_fix.js 硬编码用户 UUID 和 Supabase 凭据 中风险

问题描述：项目根目录存在调试脚本 console_sync_fix.js，包含硬编码的用户 UUID（7c9fd979-4f67-4931-ab1d-3a73ea63bbf3）和 Supabase URL + anon key。该文件未被 .gitignore 排除，会推送到远程仓库。

文件位置：console_sync_fix.js:5-8

影响：anon key 虽为公开密钥（RLS 保护），但硬编码用户 UUID 属于隐私泄露。该脚本直接操作 localStorage 和 Supabase REST API，存在被滥用的风险。

修复建议 删除 console_sync_fix.js 文件，或移至 scripts/ 目录并添加到 .gitignore。同时检查 git 历史中是否已推送该文件。
R4-S02 DOMPurify ADD_ATTR: ['*'] 配置过于宽松 低风险

问题描述：两处 DOMPurify 消毒配置使用 ADD_ATTR: ['*']（允许所有属性通过），虽然 FORBID_ATTR: ['on*'] 阻止了事件处理器，DOMPurify 默认也会过滤 javascript: URL，但此配置过于宽松。

文件位置：src/components/MermaidNodeView.vue:109 · src/components/RichTextEditor.vue:1316

实际影响：低。双重防护（FORBID_ATTR + DOMPurify 默认行为）已阻止主要攻击向量。但允许所有属性通过不符合最小权限原则。

修复建议 移除 ADD_ATTR: ['*']，改为明确列出需要的属性（如 ['d', 'x', 'y', 'cx', 'cy', 'r', 'width', 'height', 'fill', 'stroke', 'transform', 'viewBox', 'class', 'id']），或直接使用 DOMPurify 默认配置。
R4-S03 Mermaid DOMPurify 允许 style 标签（CSS 注入风险） 低风险

问题描述：MermaidNodeView.vue 第 109 行的 DOMPurify ADD_TAGS 中包含 'style' 标签，允许在 SVG 中注入 CSS。虽然 Mermaid 的 sandbox 模式和 FORBID_ATTR: ['on*'] 提供了防护，但 CSS 注入可能被用于 UI 伪装或数据泄露。

文件位置：src/components/MermaidNodeView.vue:109

实际影响：低。此为 Mermaid 渲染的功能需求（图表样式需要 <style> 标签），且内容来自 Mermaid 引擎而非直接用户输入。风险可接受。

修复建议 保留当前配置（功能需求）。如需进一步加固，可考虑用 CSP style-src 限制内联样式来源，或对 <style> 内容做正则过滤（移除 url() 和 @import）。
R4-S04 markdownExport.ts toggleToMarkdown 未转义 summary 低风险

问题描述：markdownExport.ts 第 162 行将 data-summary 属性值直接插入 <summary>${summary}</summary>，未进行 HTML 转义。summary 通过 textContent 设置（输入安全），但导出时未转义可能导致结构破坏。

文件位置：src/utils/markdownExport.ts:162

实际影响：极低。仅影响导出的 markdown 文件，不涉及应用内渲染。如果导出的 markdown 被重新导入渲染，理论上可能产生问题。

修复建议 添加 HTML 转义：summary.replace(/&/g,'&amp;').replace(/</g,'&lt;').replace(/>/g,'&gt;')
R4-S05 CSP wasm-unsafe-eval 可能不需要 低风险

问题描述：CSP 策略中包含 'wasm-unsafe-eval'，但源码中无 WebAssembly 或 wasm 直接引用。node_modules 中仅 source-map/lib/mappings.wasm 为构建依赖（非运行时）。

文件位置：index.html:7

实际影响：低。wasm-unsafe-eval 仅允许 WebAssembly 编译，不允许 eval()。但移除可减少攻击面。

修复建议 尝试移除 'wasm-unsafe-eval'，执行 npx vite build + 生产环境功能测试。如果一切正常则永久移除；如果某些依赖在运行时需要它，再保留。
