## 备忘录写入优化：Content-aware PATCH + 重试队列

针对纯在线架构下备忘录编辑的两个隐患实施优化：

### 1. Content-aware writes（性能优化）
- **问题**：800ms 防抖触发 `upsertMemo()` 时，无论哪个字段变化都发送完整 memo 对象（含 base64 图片的 content 字段可能数 MB）
- **方案**：`memoStorage.ts` 追踪 `lastWrittenContent: Map<id, content>`，当 content 未变化时改用 `supabasePatchMemo()` 发送 PATCH 请求（仅 title/tags/pinned/icon/sort_order/updated_at，跳过 content），payload 从 MB 级降至百字节级
- **新增**：`supabase.ts` 添加 `supabasePatchMemo(id, memo)` 函数，使用 PostgREST PATCH `?id=eq.xxx`

### 2. Retry queue（数据完整性）
- **问题**：fire-and-forget `supabaseUpsertMemo().catch(() => {})` 在网络中断时静默丢数据，Pinia 内存状态已更新但 Supabase 未写入，刷新后丢失
- **方案**：`pendingWrites: Map<id, {memo, retryCount}>` 队列，写入失败时自动入队，通过三个触发器重试：
  - `window.online` 事件（网络恢复）
  - `visibilitychange` 事件（标签页重新可见）
  - 30 秒定时器（周期性安全网）
- `flushPendingWrites()` 在 App.vue `handleLogout` 中 await 调用，确保登出前数据不丢

### 改动文件
- `src/services/supabase.ts`：新增 `supabasePatchMemo()`
- `src/services/memoStorage.ts`：完全重写，新增 retry queue + content tracking
- `src/App.vue`：导入 `flushPendingWrites`，`handleLogout` 改为 async

已构建并部署到 prod（三次构建零错误）。

## 紧急修复：增量同步导致页面刷新后数据为空

**根因**：P2.1 增量同步在纯在线架构中存在根本性缺陷。页面刷新后 Pinia 内存状态重置为空数组 `[]`，但 `lastSyncAt` 时间戳还在 localStorage 中。`load()` 进入增量同步分支，只获取上次同步后的更新记录，如果没有更新则 `incremental.length === 0`，数据保持空数组。且 `loaded.value = true` 后后续调用直接 return，数据永远空了。

**影响范围**：4 个 store（task / memo / todo / weeklyReport）全部受影响。

**修复方案**：移除增量同步逻辑，始终全量加载。纯在线架构没有客户端数据缓存可以合并，增量同步是过度优化。保留 `syncState.ts` 文件和 `clearLastSyncAt` 调用（清理旧时间戳），但移除 `getLastSyncAt` / `setLastSyncAt` 的使用。

**改动文件**：
- `src/stores/task.ts` — 移除增量同步，始终全量加载 tasks + trash
- `src/stores/memo.ts` — 同上
- `src/stores/todo.ts` — 同上
- `src/stores/weeklyReport.ts` — 同上

已构建并部署到 prod。
