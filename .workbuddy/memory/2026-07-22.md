## prod 部署被 git 冲突污染（紧急修复）
- 用户访问生产环境报 404 + CSP 错误。根因：**prod 仓库的 `index.html` 和 `mobile.html` 出现了未解决的 git 合并冲突标记**（`<<<<<<< HEAD` / `=======` / `>>>>>>>`），两个版本的资源引用被同时写入了文件
- 冲突来源：在我执行 robocopy 之后，有人在 prod 仓库做了 git pull/merge，产生了未解决冲突并提交了
- 报错原理：浏览器看到两组 `<meta>` 标签（CSP 报错：meta in body），同时 HTML 包含废弃 hash 名（`vue-RcofEopv.js`、`main-IXJ5LKvq.js`、`vendor-vue-BSvd92JQ.js` 等）→ 404
- 修复：
  1. 重新执行 robocopy 时加 `/IM` 标志强制覆盖（默认 robocopy 只覆盖源比目标新的文件，prod 端被外部 git 改成了更新时间，标准 robocopy 不会覆盖）
  2. 删除了 4 个不再被引用的陈旧资源文件（`vendor-vue-BSvd92JQ.js` / `_plugin-vue_export-helper-BffxEuF0.js` / `main-IXJ5LKvq.js` / `weeklyReport-DjLacL4k.js`）
- **教训**：
  - 部署脚本必须用 `robocopy ... /IM /XD .git`，否则当 prod 端有更新的外部修改时不会覆盖
  - 用户提交代码时如果遇到冲突，必须解决冲突而不是带 `<<<<<<<` 标记直接提交
  - dist 文件本身一直是干净的，问题只在 prod 仓库的 git 工作流

## TabBar 不贴底修复 + 品牌加载页恢复
- **TabBar 往上移动**：根因是 `#app` (position:fixed;inset:0;overflow:hidden) 与 `.app-shell` (position:fixed;inset:0) 嵌套。WebKit/Blink 中内层 fixed + 外层 overflow:hidden 会裁剪子元素底部。修复：`.app-shell` 改用 `height:100%;width:100%`，不重复 fixed 定位。同时恢复 TabBar 的 `box-sizing:content-box;padding-bottom:safe-area-inset` 以覆盖底部 Home 指示条。

## TabBar 贴底彻底修复（第三次）
- 前次修复（flex 布局替代 fixed）仍未解决——跨浏览器下 `#app` 内嵌套 position:fixed 元素 + #app 自身 overflow:hidden 导致 WebKit/Blink 裁剪底部
- **最终方案**：
  1. `style.css`：**去掉 `#app` 的 `overflow:hidden`**——这是导致 TabBar 底部被裁剪的根本原因。滚动穿透由 `html,body{overflow:hidden}` 兜底
  2. `MobileTabBar.vue`：**回归 `position:fixed;bottom:0;left:0;right:0`**（之前能贴底的原始方案），保留 `box-sizing:content-box` + safe-area padding
  3. `MobileApp.vue`：`.app-content` 加 `padding-bottom: calc(var(--tabbar-height) + env(safe-area-inset-bottom, 0px))`，防止内容被 fixed tabbar 遮挡
- 布局架构：`#app(position:fixed;inset:0)` → `.app-shell(height:100%;flex column)` → `.app-content(flex:1)` | `MobileTabBar(position:fixed;bottom:0)` — content 用 padding 避让 fixed tabbar
- 修改文件：`src/mobile/style.css`、`src/mobile/components/MobileTabBar.vue`、`src/mobile/MobileApp.vue`
