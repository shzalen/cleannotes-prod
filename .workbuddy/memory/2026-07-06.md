## GitHub Pages 部署配置
- 项目路由已是 `createWebHashHistory()`，无需改动
- `vite.config.ts` `base` 默认 `/`（IIS 本地），通过 `--base` CLI 参数切换
- `package.json` 新增 `build:gh-pages`：`vite build --base /cleannotes/`
- `.github/workflows/deploy.yml`：push main 时自动 `npm ci` → `npm run build:gh-pages` → peaceiris/actions-gh-pages 部署到 gh-pages 分支
- 用户需在 GitHub 仓库 Settings → Pages 中将 Source 设为 gh-pages 分支 `/ (root)`
- 手动部署方案：`scripts/deploy-gh-pages.sh` + `npm run deploy:gh-pages`：本地构建后用 git push -f 推 dist 到 gh-pages 分支，无需 GitHub Actions
- **GitHub Pages 部署坑**（cleannotes-prod 仓库）：
  - `npm run copy:prod` 构建并复制到 `D:\CleanNotepad-Prod\v1.3.0\`（仓库根目录，非子目录）
  - base 路径必须为 `/cleannotes-prod/`（仓库名即 GitHub Pages 子路径）
  - **必须添加 `.nojekyll` 文件**：否则 GitHub Pages 用 Jekyll 处理，可能跳过 JS/CSS 文件
  - `copy-to-prod.sh` 已配置每次构建自动 `touch .nojekyll`
  - 提交时必须用 `git add -A`（Vite 构建产物文件名带哈希，每次都是新文件，`git commit -am` 会漏）
  - GitHub Pages 从 `main` 分支部署，`master` 分支需通过 PR 合并到 `main`
  - 工作流：本地 master 提交 → push → GitHub PR merge 到 main → 自动触发 Pages 部署

## localStorage 跨域名隔离问题
- IIS 本地（localhost）与 GitHub Pages（shzalen.github.io）的 localStorage 完全隔离
- 部署到 GitHub Pages 后首次登录，本地无 session → 跳转登录页；登录成功后数据需从 Supabase 云端拉取
- 用户提供控制台导出/导入方案用于紧急数据迁移

## 成长系统云端同步合并逻辑修复
- **Bug**：`growthStorage.ts` `syncGrowthFromCloud()` 中 `const mergedState = { ...localState }` 直接复制本地 state，从未纳入云端数据
- 表现：Supabase 云端 level 14，本地 localStorage level 1，合并后仍为 level 1
- **修复**：改为以 `totalXp` 高的为权威源，云端 totalXp 更高时使用云端 state（streakDays 取最大值）；本地更高时 level/xp/totalXp/maxStreakDays 取最大值
- 文件：`src/services/growthStorage.ts` 第 149-164 行
