## 修复周报完成率计算

### 问题
W25 周报显示「完成 14 / 新增 14」，但完成率只有 41%。

### 根因
`src/stores/weeklyReport.ts` 中完成率公式为：
```
完成率 = 本周完成且带截止日的任务 / 所有截止日 <= 本周日的任务
```
分母把历史未完成任务也算了进去，导致数值被稀释。

### 修复
改为：
```
完成率 = 本周完成任务数 / max(本周新增任务数, 本周完成任务数)
```
用 `max` 做分母既能反映当周任务完成情况，又不会因为清理历史 backlog 出现 >100% 的异常值。

### 文件
- `src/stores/weeklyReport.ts` 第 103–114 行

## 时间戳时区 Bug 全量修复

### 问题
任务的实际开始时间（inProgressAt）和完成时间（completedAt）显示比真实时间多 8 小时。
根因：`toLocalISO()` 输出本地时间无 Z 后缀（如 `2026-07-01T12:34:00.000`），Supabase TIMESTAMPTZ 列将无时区后缀的值当作 UTC 存储。本地 CST 时间 12:34 被存为 UTC 12:34（实际应为 UTC 04:34），回读时 PG 返回 `+00` 后缀，JS 按 UTC 解析后转本地显示为 20:34。

### 修复
1. **`src/utils/time.ts`**：新增 `toUTCISO()`（= `new Date().toISOString()`，带 Z 后缀）和 `normalizeTimestamp()`（对旧无 Z 格式做本地→UTC 转换）
2. **全量替换**：所有 Store/Service 中的 `toLocalISO()` → `toUTCISO()`（task.ts/auth.ts/ai.ts/growth.ts/memo.ts/todo.ts/weeklyReport.ts/hybrid.ts/growthStorage.ts）
3. **Supabase 层防护**：`supabase.ts` 中所有 TIMESTAMPTZ 字段写入前走 `normalizeTimestamp()`，确保旧 localStorage 数据同步到 PG 时也带 Z
4. **迁移更新**：App.vue 迁移标记从 v4→v5，将旧无 Z 数据一次性转为 UTC ISO
5. **`TaskRightPanel.vue`**：`onDetailStatusChange()` 改用直接本地时间字段（不再走 `toISOString()` 再转 `toDatetimeLocal`）
6. **SQL 修复脚本**：`fix-supabase-timezone.sql`，对 PG 中历史数据减 8 小时

### 显示兼容性
`fmtTime()` / `toDatetimeLocal()` / `formatDuration()` 全部用 `new Date(ts).getHours()` 自动 UTC→本地转换，兼容新旧格式无需修改。

构建验证通过。

## 修复构建版本号不更新问题

### 问题
`npx vite build` 构建时 `__APP_VERSION__` 始终不变，`vite.config.ts` 从 `process.env.npm_package_version` 读取版本号，该变量仅在 `npm run build` 时由 npm 注入，`npx vite build` 下为 undefined。

### 修复
`vite.config.ts`：
- 新增 `appVersion()` 函数，直接从 `package.json` 读取 version 字段
- `define.__APP_VERSION__` 改用 `JSON.stringify(appVersion())` 替代 `process.env.npm_package_version`
- 额外 import `readFileSync`

构建验证通过。
