**改造内容（5 个文件）**：

   - `applyRemoteTask(remote)`：upsert 到 tasks 数组 + local storage only（不写 Supabase）

   - `applyRemoteDeletedTask(remote)`：upsert 到 trash 数组 + local only

   - 新增 `fetchRemoteChanges()` 函数：并行拉取云端+本地数据，计算差异（upserted / deletedIds）

### 删除任务后被同步还原 bug 修复 (15:50)

**问题**：删除任务后，页面暂时不显示，但过一会儿（30秒轮询同步后）删除的任务又出现了

**根因**：`fetchRemoteChanges()` 和 `mergeFromCloud()` 计算任务差异时，不交叉检查回收站表（`cleannote_deleted_tasks`），导致已删除的任务被当作"云端新增"还原回来。具体场景：
- 用户删除任务 → 本地 tasks 和 Supabase `cleannote_tasks` 删除该记录，同时写入回收站
- 如果云端删除因网络问题失败（fire-and-forget `.catch(() => {})` 静默吞错），`cleannote_tasks` 中仍保留该记录
- 30秒轮询 `fetchRemoteChanges()` 发现云端有而本地没有 → 推入 `updatedTasks` → `applyRemoteTask()` 把它加回来

**修复**（`hybrid.ts` 两处）：
1. **`fetchRemoteChanges()`**：在计算 task diff 前，先构建回收站 ID 集合 `deletedTaskIdsSet`，遍历云端任务时跳过已在回收站中的 ID；同时检测孤儿记录（同时存在于 `cleannote_tasks` 和 `cleannote_deleted_tasks`），主动清理
2. **`mergeFromCloud()`**：合并前先拉取所有数据（含回收站），构建 `deletedIdsSet`（合并云端+本地回收站 ID），过滤后再合并任务；检测并清理孤儿记录
3. **`mergeFromCloud()` 优化**：所有数据拉取合并为一次 `Promise.all`，避免分段等待

### 清空回收站后刷新任务还原 bug 修复 (16:00)

**问题**：删除任务→清空回收站→刷新页面，已删除任务又出现了

**根因**：上一个 bug 修复依赖 `cleannote_deleted_tasks` 表作为"删除证据"，但清空回收站后该记录也被删除，证据消失。如果 `cleannote_tasks` 中因云端删除失败残留了孤儿记录，`mergeFromCloud/fetchRemoteChanges` 无法识别该任务已被删除，将其当作有效数据还原。

**修复**（三层防护）：
1. **防御性清理**：`permanentDelete`/`emptyTrash`/`purgeExpired` 时同时调用 `deleteTaskById(id)`，确保 `cleannote_tasks` 云端也被清理
2. **localStorage 墓碑集合**：永久删除时将任务 ID 记入 `cleannotes_tombstones_{userId}`（格式 `{id: timestamp}`），90天自动清理。`mergeFromCloud/fetchRemoteChanges` 检查墓碑集合，跳过已删除任务；`applyRemoteTask/applyRemoteDeletedTask` 同样检查墓碑
3. **回收站显示过滤**：`loadTrash()` 过滤掉墓碑 ID，防止永久删除的项目重新出现在回收站

新增 `hybrid.ts` 导出函数：`getTombstones()`, `addTombstones()`, `removeTombstones()`
`restoreTask()` 调用 `removeTombstones()` 移除墓碑，避免恢复后又被同步过滤
